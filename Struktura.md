# Тема дипломної роботи

*Комплексний аудит безпеки та hardening типового веб-сервера на Ubuntu 24.04 LTS (Nginx/Apache, MySQL, FTP, SSH): виявлення вразливостей, моделювання загроз та автоматизовані контрзаходи.*

## Вступ

- Обґрунтування актуальності теми.
- Зв'язок роботи з практичними потребами захисту типових веб-серверів на Ubuntu 24.04 LTS.
- Огляд існуючих підходів до hardening-у: ручне налаштування, використання Ansible/Puppet, застосування CIS Benchmarks, готові security-рішення для Linux-серверів.
- Об'єкт дослідження.
- Предмет дослідження.
- Мета роботи.
- Задачі роботи:
  1. Проаналізувати архітектурні особливості Ubuntu 24.04 LTS та типового серверного стеку.
  2. Виконати моделювання загроз за методологією STRIDE для SSH, веб-сервера, MySQL та FTP.
  3. Обґрунтувати вибір інструментів аудиту та метрик оцінювання безпеки.
  4. Спроєктувати архітектуру автоматизованого інструменту hardening-у.
  5. Реалізувати Bash-скрипт автоматизації захисних налаштувань із механізмами fault tolerance.
  6. Перевірити ефективність запропонованого рішення шляхом порівняльного аудиту системи до і після hardening-у.
- Методи дослідження.
- Практичне значення отриманих результатів.

- Базові механізми захисту в Ubuntu.
- Актуальність Ubuntu 24.04 LTS для задач веб-хостингу.

#### 1.2.1. Мережевий доступ та віддалене адміністрування через SSH

#### 1.2.2. Рівень веб-сервісів: Nginx та Apache 2.4

#### 1.2.3. Рівень збереження даних: MySQL / MariaDB

#### 1.2.4. Протоколи передавання файлів: FTP / FTPS та їхні ризики

### 1.3. Моделювання загроз за методологією STRIDE

## Розділ 2. Методологія інструментального аудиту безпеки

### 2.1. Вибір стратегії аудиту та обґрунтування підходу

- Порівняння authenticated та non-authenticated scanning.
- Обґрунтування доцільності локального аудиту для задач дослідження.

### 2.2. Обґрунтування вибору інструментарію для первинного та фінального аудиту

- Використання Lynis для локального аналізу конфігурації операційної системи.
- Використання Nmap для зовнішнього мережевого сканування.
- Використання OpenVAS або аналогічного сканера для перевірки наявності відомих вразливостей.

### 2.3. Формування еталонного профілю безпеки та визначення метрик ефективності

- Поняття baseline-конфігурації.
- Hardening Index як узагальнений індикатор захищеності.
- Визначення критеріїв оцінювання результативності автоматизованого hardening-у.

### 2.4. Межі та обмеження методології дослідження

- Використання тестового середовища замість production-інфраструктури.
- Обмеження, пов'язані з конфігурацією конкретної віртуальної машини.
- Межі застосовності отриманих результатів для інших серверних сценаріїв.

## Розділ 3. Проєктування, архітектура та реалізація системи автоматизованого hardening-у

### 3.1. Обґрунтування вибору Bash як основи автоматизації

- Порівняння Bash і Ansible з точки зору простоти розгортання, залежностей і порогу входу.
- Переваги Bash для локального автономного застосування на типовому сервері.
- Обмеження Bash-підходу та способи їх компенсації.

### 3.2. Проєктування архітектури автоматизованого інструменту

- Модуль визначення активних сервісів.
- Модуль внесення безпекових змін.
- Модуль резервного копіювання конфігурацій.
- Модуль перевірки результату застосування змін.

### 3.3. Розробка механізмів відмовостійкості та захисту від помилок

- Реалізація Dead Man's Switch для безпечної зміни SSH-конфігурації.
- Попередня перевірка синтаксису конфігураційних файлів (`nginx -t`, `sshd -t`).
- Безпечний перехід від парольної автентифікації до SSH-ключів.
- Логіка rollback у разі критичної помилки.

### 3.4. Логіка журналювання та контролю виконаних дій

- Ведення лог-файлу скрипта.
- Фіксація змінених параметрів і виконаних команд.
- Забезпечення можливості подальшої перевірки та аналізу застосованих контрзаходів.

### 3.5. Модуль hardening-у SSH

- Директиви, що змінюються: `PermitRootLogin`, `PasswordAuthentication`, `MaxAuthTries`, `LoginGraceTime`, `AllowUsers`, `X11Forwarding`, `AllowAgentForwarding`, `ClientAliveInterval`/`ClientAliveCountMax`.
- Зміна стандартного порту SSH.
- Реалізація через `sed` з обов'язковою перевіркою `sshd -t` та огортанням у Dead Man's Switch.

### 3.6. Модуль hardening-у веб-сервера

- Приховування версії сервера (`server_tokens off` / `ServerTokens Prod`).
- Додавання HTTP Security Headers: `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Content-Security-Policy-Report-Only`, `Permissions-Policy`.
- Вимкнення directory listing.
- Підтримка Nginx і Apache через єдину функцію з розгалуженням за `detect_webserver()`.

### 3.7. Модуль hardening-у MySQL

- Обмеження мережевого прослуховування через `bind-address = 127.0.0.1`.
- Програмний еквівалент `mysql_secure_installation`: видалення анонімних користувачів, тестової БД, заборона віддаленого root-входу.

### 3.8. Модуль hardening-у FTP

- Увімкнення FTPS: `ssl_enable`, `force_local_logins_ssl`, `force_local_data_ssl`, обмеження до TLS 1.2+.
- Ізоляція користувачів через `chroot_local_user`.
- Обмеження пасивного порт-діапазону (`pasv_min_port=40000` / `pasv_max_port=40099`).
- Автоматичне визначення або генерація TLS-сертифіката.

### 3.9. Модуль периметра: UFW та Fail2Ban

- Стратегія default deny для вхідного трафіку; відкриття лише SSH, HTTP, HTTPS і PASV-діапазону.
- Налаштування `jail.local` для Fail2Ban: jail для SSH, vsftpd, nginx-http-auth.
- Динамічне включення jail-ів залежно від наявних сервісів.

## Розділ 4. Верифікація ефективності та аналіз результатів автоматизованого hardening-у

### 4.1. Конфігурація тестового середовища

- Характеристики віртуальної машини (RAM, CPU, диск).
- Версія Ubuntu 24.04 LTS, версії встановлених пакетів (OpenSSH, Nginx/Apache, MySQL, vsftpd).
- Спосіб розгортання (VirtualBox/VMware/cloud).

### 4.2. Результати первинного аудиту (baseline)

- Hardening Index Lynis до hardening-у.
- Кількість Warnings і Suggestions у звіті Lynis.
- Результати `nmap -sV -sC` по цільових портах.
- Ключові знахідки первинного аудиту.

### 4.3. Процес виконання hardening-у

- Запуск скрипта та приклад виводу лог-файлу.
- Опис модулів, що спрацювали під час виконання.

### 4.4. Результати фінального аудиту

- Hardening Index після hardening-у.
- Нові показники Warnings і Suggestions у звіті Lynis.
- Повторний Nmap: закриття порту 3306 ззовні, приховування SSH banner, поява HTTP security headers.

### 4.5. Порівняльний аналіз результатів

- Зведена таблиця метрик «до / після» (відповідає метрикам підрозділу 2.3).
- Аналіз змін, що дали найбільший приріст Hardening Index.
- Визначення того, що залишилось поза scope дослідження.
- Межі повної автоматизації hardening-у.
