# Розділ 4. Практична перевірка: порівняльний аудит до та після hardening-у

## 4.1. Конфігурація тестового середовища

Для практичної перевірки ефективності розробленого інструменту було розгорнуто ізольоване тестове середовище на основі технології віртуалізації. Вибір ізольованого середовища є принциповим: він дозволяє відтворити стан "свіжовстановленого VPS" без сторонніх змін, а деструктивні операції hardening-у (скидання правил UFW, перезапуск SSH) не несуть ризиків для продуктивної інфраструктури.

### Платформа віртуалізації

Тестове середовище розгорнуто у **VMware Workstation Pro 17** на хост-машині під керуванням Windows 11. VMware Workstation Pro із версії 17.5.1 (травень 2024) є безкоштовним для особистого та некомерційного використання, що робить цей вибір відтворюваним без ліцензійних витрат.

### Образ операційної системи

Використано офіційний ISO-образ **Ubuntu Server 24.04.4 LTS (Noble Numbat)**, отриманий з офіційного репозиторію Canonical:

```
https://releases.ubuntu.com/24.04/ubuntu-24.04.4-live-server-amd64.iso
SHA256: (перевірено через ubuntu.com/download/server#verify)
```

Обрано редакцію **Server** (без графічного інтерфейсу) як найбільш репрезентативну для цільового класу серверів, описаних у Вступі. Під час інсталяції активована опція **Minimized installation** для отримання стану системи, максимально наближеного до дефолтного VPS-образу хмарного провайдера.

### Конфігурація віртуальної машини

Параметри VM обрані для відповідності типовому entry-level VPS на хмарних платформах (AWS t3.small, DigitalOcean Droplet 2 GB, Hetzner CX22):

| Параметр | Значення |
|---|---|
| Пам'ять (RAM) | 2048 MB |
| Процесор | 2 vCPU |
| Диск | 20 GB (SCSI, thin provisioning) |
| Мережевий адаптер 1 | NAT (доступ до Інтернету для `apt`) |
| Мережевий адаптер 2 | Host-Only (для Nmap-сканування з хост-машини) |

Два мережевих адаптери забезпечують ізоляцію ролей: NAT-адаптер використовується виключно для завантаження пакетів під час розгортання стеку, тоді як Host-Only-адаптер є цільовою поверхнею для інструментального аудиту. Роль зовнішнього сканера виконувала окрема VM з **Kali Linux 2024.x** у тій самій Host-Only мережі (`192.168.253.0/24`). Цільовий сервер отримав адресу `192.168.253.137`. Така конфігурація відтворює типову ситуацію зовнішнього сканування сервера з незалежної машини.

### Встановлений програмний стек

Після базового розгортання ОС на систему встановлено повний серверний стек вручну — без будь-яких налаштувань безпеки, щоб зафіксувати реальний "нульовий стан" системи. Версії пакетів відповідають стандартному репозиторію Ubuntu 24.04.4 LTS на момент проведення тестування:

| Компонент | Пакет | Версія |
|---|---|---|
| Операційна система | Ubuntu Server 24.04.4 LTS | ядро 6.8.0-xx-generic |
| SSH-сервер | openssh-server | 1:9.6p1-3ubuntu13.16 |
| Веб-сервер | nginx | 1.24.0 (Ubuntu) |
| СУБД | mariadb-server | 1:10.11.x-0ubuntu0.24.04 |
| FTP-сервер | vsftpd | 3.0.5-0ubuntu2 |
| Аудит безпеки | lynis | 3.x.x |
| Мережевий сканер (Kali VM) | nmap | 7.98 |

> **Примітка щодо стану патчів у Ubuntu 24.04.4:** у поточному офіційному образі Ubuntu 24.04.4 LTS обидва компоненти — OpenSSH і MariaDB — вже містять security-виправлення: `openssh-server 1:9.6p1-3ubuntu13.16` включає патч від CVE-2024-6387 (виправлення з'явилось у `3ubuntu13.3`), а MariaDB 10.11.x у цій гілці не має публічно відомих критичних CVE на момент тестування. Тобто технічно "вразливий базовий стан" у свіжовстановленій Ubuntu 24.04.4 стосується не старих CVE, а виключно **конфігурації**: відкриті порти, відсутній firewall, дефолтні облікові записи. Проте це не знімає практичної актуальності задачі — хостери і хмарні провайдери не завжди надають образи з останніми оновленнями. AWS AMI, DigitalOcean Droplet та Hetzner Cloud часто базуються на знімку, зробленому за кілька місяців до актуального релізу Canonical, і не гарантують автоматичного застосування `apt upgrade` перед першим завантаженням. Адміністратор, який не виконав `apt update && apt upgrade` одразу після розгортання, може опинитися з непропатченим пакетом — саме тому hardening-скрипт виконує перевірку та оновлення пакетів як перший крок.

### Стан системи перед hardening-ом

Після встановлення всіх компонентів стек не отримував жодних налаштувань безпеки. Цей стан є відправною точкою для первинного аудиту (підрозділ 4.2) і безпосередньо відповідає сценарію, описаному у Вступі: сервер встановлено і виведено в мережу з дефолтною конфігурацією.

Ключові характеристики початкового стану:
- SSH доступний на стандартному **порту 22**, дозволено вхід під `root` з паролем;
- MariaDB за замовчуванням прив'язана до `127.0.0.1` (порт 3306 недоступний ззовні) — однак конфігурація нічим не зафіксована і може бути змінена випадково;

- FTP працює без шифрування (TLS вимкнено за замовчуванням у vsftpd);
- Nginx не встановлює жодних security-заголовків, версія сервера розкривається у відповідях;
- UFW встановлений, але **вимкнений** (`Status: inactive`) — жодних правил фільтрації не застосовано.

## 4.2. Результати первинного аудиту (baseline)

Первинний аудит проводився двома незалежними інструментами: Lynis виконував аудит системи зсередини VM, тоді як Nmap виконував зовнішнє сканування з Kali Linux у тій самій Host-Only мережі. Ця двошарова методологія дозволяє оцінити систему як з точки зору внутрішніх слабкостей конфігурації, так і з точки зору зовнішньої поверхні атаки, яку бачить потенційний зловмисник.

### 4.2.1. Результати аудиту Lynis

Команда запуску:

```bash
sudo lynis audit system 2>&1 | tee ~/lynis_before.txt
```

**Підсумкові показники:**

| Метрика | Значення |
|---|---|
| Hardening index | **63** / 100 |
| Виконано тестів | 261 |
| Warnings | **1** |
| Suggestions | **39** |

**Єдиний Warning — `FIRE-4512`:**

```
! iptables module(s) loaded, but no rules active [FIRE-4512]
```

UFW встановлений у системі, але перебуває у вимкненому стані. Iptables-модулі ядра завантажені, але жодних правил фільтрації не застосовано — система повністю відкрита на мережевому рівні.

**Ключові Suggestions у межах scope hardening-скрипта:**

| Код | Опис | Дія скрипта |
|---|---|---|
| `DEB-0880` | Встановити fail2ban | ✓ встановлює і налаштовує |
| `FIRE-4512` | Немає активних правил firewall | ✓ UFW default deny + правила |
| `HTTP-6710` | Додати HTTPS до nginx | поза scope (потребує домен) |
| `BANN-7126/7130` | Legal banners у /etc/issue | поза scope |
| `KRNL-6000` | Налаштування sysctl | поза scope |
| `HRDN-7230` | Встановити malware scanner | поза scope |
| `ACCT-9628` | Увімкнути auditd | поза scope |

З 39 suggestions лише 2 безпосередньо усуваються скриптом hardening-у. Решта стосуються категорій, що виходять за межі заявленого scope: налаштування ядра, окремі розділи файлової системи, системи виявлення вторгнень, управління паролями PAM. Це нормальна ситуація: hardening-скрипт вирішує найбільш критичні проблеми для типового веб-стеку, але не претендує на повне виконання всіх 200+ пунктів CIS Benchmark.

**Аналіз результатів systemd-analyze security:**

Lynis виявив, що більшість системних сервісів мають статус `UNSAFE` за шкалою ізоляції systemd. Це системна характеристика Ubuntu 24.04 — сервіси з дистрибутива не мають повного набору `SystemCallFilter`, `PrivateTmp`, `ProtectSystem` та інших директив sandboxing. Зокрема:

- `nginx.service` — `UNSAFE`
- `vsftpd.service` — `UNSAFE`
- `ssh.service` — `UNSAFE`
- `mariadb.service` — `EXPOSED` (частково ізольований)

Зміцнення systemd unit-файлів виходить за межі scope цієї роботи, але є природним напрямком розвитку інструменту.

### 4.2.2. Результати зовнішнього сканування Nmap

Сканування виконувалось з Kali Linux (`192.168.253.1`) проти цільового сервера (`192.168.253.137`).

```bash
# Повне сканування з визначенням версій і NSE-скриптами
nmap -sV -sC 192.168.253.137

# Цільове сканування по портах стеку
nmap -sV -p 22,80,443,21,3306 192.168.253.137
```

**Результати сканування:**

| Порт | Стан | Сервіс | Версія | Проблема |
|---|---|---|---|---|
| 21/tcp | open | ftp | vsftpd 3.0.5 | Без TLS — паролі відкритим текстом |
| 22/tcp | open | ssh | OpenSSH 9.6p1 Ubuntu 3ubuntu13.16 | Порт 22, повна версія в banner; CVE-2024-6387 |
| 80/tcp | open | http | nginx 1.24.0 (Ubuntu) | Версія і ОС у заголовку `Server` |
| 443/tcp | closed | — | — | HTTPS не налаштований |
| 3306/tcp | closed | — | — | MariaDB вже прив'язана до localhost |

SSH-banner (`OpenSSH 9.6p1 Ubuntu 3ubuntu13.16`) розкриває точну версію пакета разом із суфіксом дистрибутива Ubuntu. Хоча версія `3ubuntu13.16` вже містить backported-патч від CVE-2024-6387 (виправлення з'явилось у `3ubuntu13.3`), відкритий рядок версії дозволяє зловмиснику точно встановити пакет і дистрибутив — що є відправною точкою для пошуку інших, ще невиправлених вразливостей, специфічних для даної конфігурації. Саме тому приховування banner-у є частиною SSH hardening-у незалежно від патч-рівня.

Заголовок `Server: nginx/1.24.0 (Ubuntu)` у HTTP-відповіді розкриває не лише версію веб-сервера, а й дистрибутив операційної системи — інформація, що спрощує цілеспрямований пошук вразливостей для конкретної конфігурації.

Закритий порт 3306 підтверджує, що MariaDB на Ubuntu 24.04 за замовчуванням прослуховує лише на loopback-інтерфейсі. Це є покращенням порівняно з попередніми версіями Ubuntu, де MySQL прослуховував на всіх інтерфейсах. Проте внутрішній стан бази даних (анонімні користувачі, тестова БД, дозволи) залишається незахищеним і потребує окремого модуля.
