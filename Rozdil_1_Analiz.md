# Розділ 1. Аналіз вразливостей та моделювання загроз типового веб-сервера

## 1.1. Архітектурні особливості та безпекові можливості Ubuntu 24.04 LTS

Ubuntu 24.04 LTS вийшов у квітні 2024 року під кодовою назвою Noble Numbat і отримає оновлення безпеки від Canonical до квітня 2029 року. Це не просто черговий реліз, а свідомо найбільш захищена версія Ubuntu на момент виходу. Canonical зробила кілька принципових архітектурних рішень, які суттєво змінюють поверхню атаки порівняно з попередніми версіями.

### Що нового в безпеці

**Ядро Linux 6.8 із розширеним binary hardening.** FORTIFY_SOURCE підвищено до рівня 3, що означає жорсткішу перевірку меж пам'яті при виклику стандартних функцій на кшталт memcpy() та snprintf(). Простою мовою: певний клас помилок із пам'яттю тепер виявляється і блокується ще на рівні ядра, не доходячи до рівня застосунку.

**AppArmor 4.0.** У цій версії AppArmor отримав підтримку умовних політик доступу і можливість делегувати рішення про доступ до застосунків у реальному часі. На практиці це означає більш точні профілі обмежень для процесів, що запускаються на сервері.

**Обмеження непривілейованих user namespaces.** Це, мабуть, найважливіша безпекова зміна в Ubuntu 24.04. User namespaces - механізм ізоляції Linux, який дає змогу програмі думати, що вона запущена з правами root, залишаючись при цьому непривілейованою з точки зору реального ядра. Ця функція активно використовується контейнерами, браузерами та sandbox-середовищами, але за нею тягнеться довга історія вразливостей ядра, які вона відкривала. Ubuntu 24.04 вперше за замовчуванням обмежує доступ до user namespaces через AppArmor. Варто чесно відзначити, що в січні 2025 року дослідники Qualys виявили три обхідні шляхи цього механізму через стандартні утиліти (aa-exec, busybox та LD_PRELOAD). Canonical кваліфікувала їх не як вразливості, а як обмеження поточної реалізації, і пообіцяла посилити захист у наступних релізах. Цей випадок добре ілюструє загальну ідею: навіть добре продумані захисні механізми потребують регулярного перегляду і не можуть бути єдиним рубежем оборони.

**Відключення застарілих версій TLS.** Ubuntu 24.04 за замовчуванням вимикає TLS 1.0, TLS 1.1 та DTLS 1.0. Ці версії протоколу давно вважаються небезпечними через ряд відомих атак на зниження рівня шифрування, але продовжували бути доступними в попередніх релізах для забезпечення сумісності.

### Що Ubuntu не робить за вас

Тут починається найважливіше для цієї роботи. Попри всі покращення, Ubuntu 24.04, як і будь-який інший дистрибутив, орієнтується на максимальну сумісність при першому запуску - а не на максимальний захист. Конкретні наслідки цього:

- SSH встановлюється з підтримкою парольної автентифікації та, залежно від конфігурації, може дозволяти вхід під root.
- Nginx і Apache встановлюються без жодних обмежень на розкриття інформації про сервер.
- MySQL і vsftpd при стандартній установці не мають жодних мережевих обмежень понад базові дефолти пакета.
- UFW встановлено у систему, але за замовчуванням він вимкнений.

Тобто Ubuntu 24.04 дає адміністратору сильний фундамент із хорошими вбудованими механізмами - але не звільняє його від необхідності правильно налаштувати сервіси, які на цьому фундаменті запущені.

## 1.2. Аналіз базового технологічного стеку веб-сервера

Типовий веб-сервер на Ubuntu складається з чотирьох ключових компонентів: протоколу віддаленого управління (SSH), веб-сервера (Nginx або Apache), бази даних (MySQL або MariaDB) і засобу передачі файлів (FTP/FTPS). Кожен із них має власний набір ризиків, які виникають не обов'язково через програмні вразливості, а часто просто через типові, нічим не примітні дефолтні конфігурації.

### 1.2.1. Мережевий доступ та віддалене адміністрування через SSH

SSH - це перші ворота на сервер, і саме тому він привертає найбільшу увагу атакуючих. Протокол сам по собі є криптографічно сильним і добре продуманим, проблема виключно в тому, як він налаштований.

Ubuntu 24.04 постачається з OpenSSH версії 9.6p1. У стандартній конфігурації:

- SSH слухає на порту 22, який є першим у черзі для будь-якого автоматизованого сканера.
- Дозволена парольна автентифікація, що відкриває двері для brute-force атак.
- Залежно від версії та конфігурації може бути дозволений вхід під користувачем root.
- Banner і версія OpenSSH видаються клієнту до автентифікації.

### 1.2.2. Рівень веб-сервісів: Nginx та Apache 2.4

Nginx і Apache - два найпоширеніші веб-сервери для типового LAMP/LEMP-стеку. З точки зору безпеки вони схожі в своїх дефолтних проблемах, але відрізняються в деталях конфігурації.

**Розкриття технічної інформації.** І Nginx, і Apache за замовчуванням вказують свою версію в HTTP-заголовку Server кожної відповіді. Nginx іде ще далі і вказує версію навіть на сторінках помилок 404 і 403. Для атакуючого це безкоштовна розвідка: знаючи точну версію, можна миттєво перевірити список відомих CVE для неї. Конфігурація server_tokens off (Nginx) або ServerTokens Prod (Apache) вирішує проблему однією строчкою, але за замовчуванням не встановлена. Цікаво, що CIS Benchmark включає перевірку цього параметра як обов'язкову - тобто галузевий стандарт вважає це базовою вимогою, а не опцією.

### 1.2.3. Рівень збереження даних: MySQL / MariaDB

MySQL і MariaDB у типовому веб-хостингу використовуються як backend для CMS, інтернет-магазинів та будь-яких веб-застосунків із базою даних. Їхні дефолтні конфігурації несуть два принципово різні класи ризиків.

**Перший клас - мережева доступність.** Дослідження Shadowserver Foundation виявило у відкритому Інтернеті понад 3,6 мільйона MySQL-серверів, які реагують на з'єднання по порту 3306/TCP. Переважна більшість із них не потребує зовнішнього доступу - вони просто мають дефолтну конфігурацію, де MySQL слухає на всіх інтерфейсах замість 127.0.0.1. Для типового веб-сайту MySQL взагалі не повинен бути доступним ззовні: вебсервер і база даних знаходяться на одній машині і спілкуються через localhost. Одна строчка bind-address = 127.0.0.1 у конфігурації закриває цілий клас зовнішніх атак без жодних функціональних втрат.

**Другий клас - стан після інсталяції.** Навіть якщо порт закритий, типова установка MySQL може залишати: тестову базу даних test, анонімного користувача, порожній пароль для root-облікового запису та root-доступ без обмеження хоста. Утиліта mysql_secure_installation існує саме для того, щоб прибрати всі ці "подарунки за замовчуванням", але вона не запускається автоматично при установці і вимагає ручного виконання.

### 1.2.4. Протоколи передавання файлів: FTP / FTPS та їхні ризики

FTP - протокол, якому понад 50 років, і він принципово небезпечний у своєму класичному вигляді. Річ у тому, що FTP передає і ім'я користувача, і пароль відкритим текстом у мережі. Будь-хто, хто може прослуховувати трафік між клієнтом і сервером (наприклад, в одній мережі або при відсутності шифрування на проміжних ланках), отримує облікові дані без жодних зусиль.

vsftpd, стандартний FTP-сервер для Ubuntu, за замовчуванням не вмикає TLS і не ізолює користувачів у їхніх домашніх директоріях через chroot. Це означає: автентифікація відкритим текстом і потенційна можливість для користувача вийти за межі призначеної директорії. Обидві проблеми вирішуються конфігурацією, але не вирішуються самостійно.

## 1.3. Моделювання загроз за методологією STRIDE

STRIDE - це класифікаційна схема загроз, розроблена в Microsoft, яка розбиває всі можливі атаки на шість категорій: Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service та Elevation of Privilege. Ця схема зручна тим, що змушує системно пройтися по кожному компоненту і задати собі питання: а що саме тут може піти не так і в яку категорію це потрапляє?

### Матриця загроз STRIDE

| Компонент | Категорія STRIDE | Загроза | Контрзахід |
|---|---|---|---|
| SSH | Information Disclosure | Розкриття версії OpenSSH через banner до автентифікації | DebianBanner no, обмеження pre-auth інформації |
| SSH | Elevation of Privilege | Brute-force парольної автентифікації, в т.ч. під root | PermitRootLogin no, PasswordAuthentication no, Fail2Ban |
| SSH | Spoofing | Підміна ключів при першому підключенні (TOFU-атака) | Завчасне додавання відбитку ключа в known_hosts |
| SSH | Denial of Service | Перевантаження сервера з'єднаннями | MaxStartups, LoginGraceTime, rate-limiting UFW |
| Web (Nginx/Apache) | Information Disclosure | Розкриття версії сервера в заголовку Server | server_tokens off / ServerTokens Prod |
| Web (Nginx/Apache) | Tampering | XSS-атаки через відсутність CSP | Content-Security-Policy: default-src 'self' (Report-Only) |
| Web (Nginx/Apache) | Information Disclosure | Перегляд вмісту директорій через directory listing | autoindex off / -Indexes |
| Web (Nginx/Apache) | Tampering | Clickjacking через відсутність X-Frame-Options | X-Frame-Options: SAMEORIGIN |
| Web (Nginx/Apache) | Tampering | MIME-sniffing атаки | X-Content-Type-Options: nosniff |
| MySQL | Denial of Service | Зовнішні запити до порту 3306 | bind-address = 127.0.0.1 |
| MySQL | Elevation of Privilege | Анонімний доступ, порожній root-пароль | mysql_secure_installation, видалення анонімних користувачів |
| MySQL | Information Disclosure | Витік схеми БД через надмірні привілеї | Принцип мінімальних привілеїв для кожного користувача |
| FTP | Information Disclosure | Передача облікових даних відкритим текстом | Обов'язкове шифрування через FTPS (TLS) |
| FTP | Elevation of Privilege | Вихід за межі домашньої директорії | chroot_local_user=YES у vsftpd |
| FTP | Spoofing | Підміна FTP-трафіку при відсутності шифрування | Міграція на FTPS або SFTP |
| Периметр | Denial of Service | Перебір портів і автоматизоване сканування | UFW Default Deny, Fail2Ban для SSH і FTP |

## 1.4. Аналіз актуальних вразливостей і типових помилок конфігурації

Попередні розділи розглядали ризики, закладені в типові дефолтні конфігурації. Однак реальна картина загроз складається не лише з неоптимальних налаштувань: час від часу у широко розповсюджених сервісах виявляються критичні вразливості, які принципово змінюють оцінку пріоритетів. У 2024 році такою вразливістю стала CVE-2024-6387, відома під назвою regreSSHion. Її аналіз доцільно розглянути як ілюстрацію того, яким чином навіть добре написане програмне забезпечення може отримати критичний вектор атаки — і чому своєчасний hardening та патч-менеджмент мають рівну вагу.

### 1.4.1. CVE-2024-6387 (regreSSHion): механізм, вектор атаки та масштаб поширення

**Відкриття та базові відомості.** У липні 2024 року дослідники Qualys оприлюднили звіт про критичну вразливість в OpenSSH, якій було присвоєно ідентифікатор CVE-2024-6387 та неофіційну назву regreSSHion. Вразливість отримала оцінку CVSS 8.1 (High) і дозволяє неавтентифікованому зловмиснику виконати довільний код на сервері з правами root. Для успішної атаки не потрібні ані облікові дані, ані взаємодія з легітимним користувачем — достатньо мережевого доступу до порту SSH.

**Механізм вразливості.** regreSSHion є race condition у обробнику сигналу SIGALRM у серверному процесі sshd. Коли клієнт ініціює TCP-з'єднання, але не завершує автентифікацію протягом інтервалу `LoginGraceTime` (за замовчуванням 120 секунд), ядро посилає процесу сигнал SIGALRM. Обробник цього сигналу викликає функцію `syslog()` — і тут полягає проблема: `syslog()` не є async-signal-safe функцією за стандартом POSIX. Виклик async-signal-unsafe функції з обробника сигналу у будь-який момент виконання основного потоку означає, що стан внутрішніх структур libc може бути частково модифікований, коли сигнал перерве основний потік.

Зловмисник може навмисно синхронізувати моменти надсилання з'єднань і переривання таким чином, щоб досягти пошкодження структур купи (heap corruption) у адресному просторі sshd. Система розподілу пам'яті glibc містить метадані прямо в блоках купи, і цілеспрямоване їх пошкодження з наступним контрольованим розігруванням ситуації дозволяє досягти виконання довільного коду. На системах із glibc (до яких належить Ubuntu) атака технічно реалізовна; на системах із musl libc (Alpine Linux та ін.) вразливість присутня, але її практична експлуатація суттєво ускладнена.

**Чому «regreSSHion».** Назва вразливості — це каламбур: regression + SSH. Справа в тому, що ця проблема вже існувала раніше: у 2006 році вона була зафіксована як CVE-2006-5051 і виправлена в OpenSSH 4.4p1. Однак у 2020 році, при рефакторингу коду обробки сигналів у версії 8.5p1, виправлення було випадково видалено — і вразливість повернулась. Таким чином, уразливими є всі версії OpenSSH від 8.5p1 до 9.7p1 включно. Версії, старіші за 4.4p1, також вразливі, якщо не мають патча CVE-2006-5051, але їх наявність у продуктивних середовищах у 2024 році є практично нульовою. Виправлення увійшло до складу OpenSSH 9.8p1, випущеного 1 липня 2024 року.

**Ubuntu 24.04 LTS і ця вразливість.** Ubuntu 24.04 LTS вийшов у квітні 2024 року з OpenSSH версії 9.6p1 — тобто у вразливому діапазоні. Canonical випустила виправлення у складі пакета оновлень безпеки 1 липня 2024 року, одночасно з публічним розкриттям. Сервери, на яких не застосовувались оновлення безпеки, залишались вразливими.

**Умови та складність експлуатації.** Атака є стохастичною за своєю природою: вона ґрунтується на виграші race condition, ймовірність якого за один цикл є низькою. За оцінками Qualys, для успішної експлуатації зловмисник у середньому повинен зробити від кількох тисяч до декількох десятків тисяч спроб з'єднання. На практиці це означає тривалий шум у мережі і помітне навантаження на сервер, що полегшує виявлення атаки. Крім того, атака не є детерміністичною: кількість необхідних спроб залежить від конкретного стану пам'яті процесу і може суттєво варіюватись. На момент публічного розкриття вразливості жодного публічного proof-of-concept для серверів x86-64 не існувало, однак Qualys підтвердила успішну демонстрацію на тестових 32-бітних системах.

**Масштаб поширення.** Qualys оцінила кількість потенційно вразливих серверів у відкритому Інтернеті приблизно в 14 мільйонів. Це серверна частина всього встановленого парку OpenSSH відповідних версій. З них реально вразливими є ті, що відповідають двом умовам: вони не отримали патч безпеки від дистрибутива і слухають SSH на відкритому мережевому інтерфейсі без обмежень доступу.

**Наслідки для hardening-у.** regreSSHion ілюструє три принципові висновки для цієї роботи. По-перше, регулярне застосування оновлень безпеки є обов'язковою, а не опціональною частиною захисту сервера. По-друге, навіть за відсутності патча, ряд hardening-заходів знижує ризик: зміна порту SSH зменшує автоматизоване сканування, Fail2Ban блокує масові спроби з'єднання (хоча й не є панацеєю при розподіленій атаці), `MaxStartups` обмежує одночасні неавтентифіковані з'єднання і тим самим ускладнює виграш race condition. По-третє, LoginGraceTime є безпосереднім параметром, що впливає на часове вікно атаки: зменшення його значення зі 120 до 30 секунд скорочує доступний зловмиснику інтервал, хоча й не усуває вразливість.

### 1.4.2. Типові ризики дефолтних конфігурацій: зведений аналіз

Попри всю гучність CVE-2024-6387, переважна більшість успішних атак на реальні веб-сервери відбувається не через zero-day вразливості, а через абсолютно передбачувані наслідки неналежної конфігурації. Нижче наведено зведений аналіз ризиків для кожного компоненту стеку.

**SSH.** Дефолтна конфігурація OpenSSH дозволяє парольну автентифікацію, що робить сервер мішенню для автоматизованих brute-force атак. Статистика Shodan і звіти Fail2Ban-спільноти стабільно показують, що сервер на дефолтному порту 22 починає отримувати спроби підбору облікових даних протягом кількох хвилин після підключення до Інтернету — без будь-яких цілеспрямованих дій з боку зловмисника. Розкриття версії OpenSSH через banner до автентифікації дозволяє атакуючому негайно звузити перелік релевантних CVE. Дозволений вхід під root означає, що успішний brute-force або підбір ключа одразу дає повний контроль над системою без необхідності подальшого підвищення привілеїв.

**Nginx та Apache.** Розкриття точної версії сервера в заголовку `Server` перетворює кожну HTTP-відповідь на безкоштовний розвідувальний звіт для атакуючого. Відсутність HTTP security headers залишає користувачів браузерів уразливими до XSS, clickjacking та MIME-sniffing атак навіть при правильно написаному серверному коді. Directory listing, увімкнений за замовчуванням у деяких конфігураціях Apache, може розкрити структуру файлової системи, резервні копії або тимчасові файли, що не повинні бути публічно доступними.

**MySQL.** Прослуховування на всіх мережевих інтерфейсах (0.0.0.0:3306) робить базу даних доступною ззовні — при цьому для типового веб-сайту зовнішній доступ до MySQL просто не потрібен. Разом із можливим порожнім паролем root або анонімним доступом це утворює готовий вектор для несанкціонованого доступу до всіх даних застосунку. Наявність тестової бази `test` і анонімного користувача є прямим наслідком дефолтної установки без виконання кроків `mysql_secure_installation`.

**vsftpd / FTP.** Передача облікових даних відкритим текстом у класичному FTP робить будь-яке перехоплення трафіку рівнозначним компрометації облікового запису. Відсутність chroot-ізоляції дозволяє автентифікованому FTP-користувачу потенційно переміщатись поза межі призначеної директорії. Обидва ці ризики є прямим наслідком дефолтної конфігурації vsftpd, яка не вмикає TLS і не встановлює `chroot_local_user=YES`.

### 1.4.3. Вплив відкритих сервісів, слабкої автентифікації та відсутності шифрування на загальний рівень безпеки

Окремо взятий ризик дефолтної конфігурації може здаватись керованим: brute-force SSH завершиться невдачею, якщо пароль достатньо складний; відкритий порт MySQL не завдасть шкоди, якщо root захищений надійним паролем. Однак у реальній ситуації ризики накладаються і посилюють один одного.

**Ефект накопичення вразливостей.** Сервер, на якому SSH дозволяє парольну автентифікацію, MySQL слухає на зовнішньому інтерфейсі, а FTP передає дані відкритим текстом, формує ширшу поверхню атаки, ніж кожна з цих проблем окремо. Зловмисник, який перехопив FTP-трафік і отримав системні облікові дані, може спробувати їх повторно для SSH. Вхід до SSH може відкрити доступ до MySQL через localhost. Таким чином, компрометація через один вектор автоматично означає ризик для всіх суміжних компонентів.

**Інформаційна асиметрія на користь атакуючого.** Кожне відкрите поле — banner SSH, заголовок Server у HTTP-відповіді, відповідь MySQL на з'єднання — надає атакуючому інформацію, яку він може використати для звуження своєї тактики. Захиснику ця інформація не потрібна ззовні: вона корисна лише для адміністрування, а не для публічного доступу. Приховування технічних деталей від зовнішнього спостерігача безпосередньо підвищує вартість розвідувальної фази атаки.

**Відсутність шифрування як системна проблема.** Використання класичного FTP без TLS у мережевому середовищі означає, що будь-який проміжний вузол між клієнтом і сервером може в реальному часі отримувати облікові дані та вміст файлів. У корпоративних мережах, де трафік може проходити через перемикачі з можливістю port mirroring, або у хмарних середовищах, де мережевий трафік між VM може логуватись інфраструктурою провайдера, ця загроза є цілком реалістичною.

**Відсутність мережевого периметру як підсилювач усіх ризиків.** UFW, встановлений але не активований, є символом ширшої проблеми: захисні механізми є, але не ввімкнені. Сервер без активного файрволу виставляє назовні всі слухаючі порти, включаючи порт MySQL 3306, адміністративні інтерфейси та потенційно тимчасово відкриті під час налагодження порти. Активація UFW з правилом default deny і явним дозволом лише необхідних портів усуває цілий клас ризиків однією операцією.

Таким чином, аналіз вразливостей і типових помилок конфігурації демонструє: результативний hardening вимагає не усунення якоїсь однієї проблеми, а системного підходу до всього стеку одночасно. Саме ця логіка визначає структуру автоматизованого скрипта, описаного в Розділі 3: кожен модуль закриває окремий вектор атаки, а разом вони формують узгоджений захисний контур.
