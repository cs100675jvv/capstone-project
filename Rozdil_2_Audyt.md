# Розділ 2. Методологія інструментального аудиту безпеки

## 2.1. Вибір стратегії аудиту та обґрунтування підходу

Перш ніж обирати конкретні інструменти, потрібно визначити сам підхід до перевірки безпеки сервера: оцінювати його ззовні чи зсередини. У практиці аудиту ці два варіанти зазвичай описують як unauthenticated scanning і authenticated scanning.

### Unauthenticated vs Authenticated: у чому різниця

Unauthenticated-сканування — це погляд на сервер з позиції стороннього користувача або зловмисника. Сканер не має облікових даних і бачить тільки те, що система готова віддати назовні: відкриті порти, банери сервісів, заголовки відповідей і реакцію на типові запити. Такий підхід добре показує, що саме доступно атакуючому ще до будь-якої спроби компрометації.

Authenticated-сканування, навпаки, дає змогу подивитися на систему зсередини. У цьому випадку утиліта має доступ до сервера і може перевірити конфігурацію, права доступу, встановлені пакети, активні процеси та параметри ядра. Це значно повніша картина, але для її отримання потрібен доступ до самої системи.

Обидва підходи корисні, однак вони відповідають на різні питання. Unauthenticated-сканування показує зовнішню поверхню атаки, а authenticated — реальний стан системи після внесених змін. Саме тому в цій роботі доцільно використовувати їх у поєднанні.

### Чому для цієї роботи обрано authenticated-підхід

Мета дослідження полягає не в тому, щоб лише імітувати зовнішню атаку, а в тому, щоб виміряти реальний ефект від застосування hardening-у. Потрібно зрозуміти, що саме змінилося в конфігурації сервера після запуску скрипта, а це неможливо повноцінно побачити лише зовні.

Крім того, частина змін узагалі не помітна без доступу до системи. Наприклад, параметр `bind-address = 127.0.0.1` для MySQL може закривати зовнішній доступ до порту, але якщо сервіс і так був прикритий фаєрволом, зовнішній сканер не покаже різниці. Так само зміна `PermitRootLogin no` у SSH не змінює набір відкритих портів, але суттєво змінює рівень захисту.

Основним інструментом authenticated-аудиту в роботі обрано Lynis, а Nmap використовується як додатковий засіб для зовнішньої перевірки мережевої поверхні до і після hardening-у.

## 2.2. Обґрунтування вибору інструментарію для первинного та фінального аудиту

### Lynis — основний інструмент аудиту

Lynis — це відкрита утиліта локального аудиту безпеки для Linux, macOS та Unix-систем. Вона запускається безпосередньо на сервері з правами root і перевіряє конфігурацію ОС, мережевих служб, прав доступу до файлів, параметрів ядра та інших аспектів безпеки. Після завершення перевірки Lynis формує звіт і розраховує Hardening Index — числовий показник від 0 до 100, який відображає поточний рівень захищеності системи.

Саме Hardening Index є головною причиною вибору Lynis у цій роботі. Він дозволяє звести оцінку безпеки до одного порівнюваного значення до і після застосування hardening-у. Це робить результат не просто описовим, а вимірюваним.

Lynis не потребує складного налаштування і запускається однією командою:

> `lynis audit system`

Для свіжої Ubuntu 24.04 без додаткової конфігурації типовим є результат у межах 50–65 балів. Після застосування базових рекомендацій безпеки показник зазвичай зростає до 70–85 балів. Саме ця різниця і використовується в роботі як практичний ефект hardening-у.

Lynis поділяє результати на кілька категорій: Warnings, Suggestions і Hardening Index. Така структура зручна тим, що окремо показує критичні проблеми, рекомендації та загальний підсумок.

### Nmap — мережеве сканування

Nmap використовується для двох завдань: перевірки зовнішньої мережевої поверхні та порівняння результатів до і після hardening-у. Базова команда для перевірки відкритих портів і версій сервісів виглядає так:

> `nmap -sV -sC -p 22,80,443,3306,21 <IP-сервера>`

Прапор `-sV` дозволяє визначати версії сервісів, а `-sC` запускає стандартний набір NSE-скриптів для кожного знайденого порту.

Для цієї роботи особливо корисні такі NSE-скрипти:

| NSE-скрипт | Що перевіряє |
|---|---|
| `ssh-hostkey` | Відбиток і тип SSH-ключів сервера |
| `ssh2-enum-algos` | Підтримувані алгоритми шифрування SSH |
| `http-server-header` | Значення заголовка `Server` у відповіді веб-сервера |
| `http-security-headers` | Наявність або відсутність заголовків безпеки |
| `mysql-empty-password` | Перевірка порожнього пароля root у MySQL |
| `ftp-anon` | Перевірка анонімного доступу до FTP |
| `ftp-syst` | Банер і версія FTP-сервера |

### OpenVAS та легковагові альтернативи: Trivy і Grype

OpenVAS — це повноцінний сканер вразливостей корпоративного рівня. На відміну від Lynis і Nmap, він орієнтується на базу відомих CVE і перевіряє, чи вразлива конкретна версія програмного забезпечення. Водночас його використання потребує окремого сервера, значних ресурсів і окремого веб-інтерфейсу. Для задачі цієї роботи такі витрати виглядають надлишковими.

Як легшу альтернативу для перевірки пакетних CVE можна використати Trivy. Це інструмент, який перевіряє пакети ОС, бібліотеки та контейнери безпосередньо на цільовій машині і не потребує складної інфраструктури.

Встановлення і запуск сканування файлової системи:

```bash
apt-get install -y wget apt-transport-https
wget -qO - https://aquasecurity.github.io/trivy-repo/deb/public.key | apt-key add -
echo "deb https://aquasecurity.github.io/trivy-repo/deb generic main" \
  > /etc/apt/sources.list.d/trivy.list
apt-get update && apt-get install -y trivy

trivy fs --scanners vuln / 2>/dev/null
```

Скорочений приклад виводу Trivy для Ubuntu 24.04 показує, що інструмент реально знаходить вразливі пакети ще до hardening-у.

Порівняння інструментів аудиту в цій роботі наведено нижче:

| Інструмент | Тип аудиту | Що перевіряє | Ресурси |
|---|---|---|---|
| Lynis | Локальний, конфігураційний | Параметри ОС, служб, прав доступу | Мінімальні |
| Nmap | Мережевий, зовнішній | Відкриті порти, версії сервісів, NSE-скрипти | Мінімальні |
| Trivy | Локальний, CVE-база | Вразливі версії встановлених пакетів | Мінімальні |
| OpenVAS | Мережевий, CVE-база | Повний CVE-аудит із веб-інтерфейсом | Значні |

У межах роботи Trivy використовується як додатковий інструмент, щоб показати, що вразливості на кшталт regreSSHion можна виявити не лише теоретично, а й на рівні перевірки встановлених пакетів.

## 2.3. Формування еталонного профілю безпеки та визначення метрик ефективності

### Що таке baseline і навіщо він потрібен

Еталонний профіль безпеки, або baseline, — це зафіксований стан системи до внесення будь-яких змін. У межах цієї роботи baseline включає результат сканування Lynis, вивід Nmap по цільових портах і стан основних конфігураційних файлів сервісів на момент першого запуску.

Без baseline неможливо коректно оцінити, наскільки сильно змінилася безпека після hardening-у. Це звичайна логіка порівняння: спочатку фіксується стартовий стан, а потім уже оцінюється різниця.

Baseline у цій роботі фіксується у три кроки: чиста установка Ubuntu 24.04 LTS зі стандартним набором сервісів, запуск `lynis audit system` із збереженням звіту та запуск `nmap -sV -sC` по цільових портах.

### Метрики ефективності hardening-у

Головною метрикою є Hardening Index Lynis. У цій роботі очікується, що після застосування автоматизованого hardening-у цей показник зросте з базового рівня приблизно 50–65 до рівня 75+.

Додаткові метрики наведені в таблиці:

| Метрика | Спосіб вимірювання | Очікуваний результат |
|---|---|---|
| Hardening Index | Lynis до/після | Зростання з 50–65 до 75+ |
| Кількість Warnings | Lynis до/після | Зменшення на 50%+ |
| Кількість Suggestions | Lynis до/після | Зменшення на 30%+ |
| Відкриті зовнішні порти | Nmap до/після | Порт 3306 зникає ззовні |
| SSH banner | Nmap `ssh-hostkey` | Версія OpenSSH прихована |
| HTTP Security Headers | Nmap `http-security-headers` | З’являються HSTS, X-Frame-Options |
| FTP анонімний доступ | Nmap `ftp-anon` | Відмова у доступі |

### Як інтерпретувати Hardening Index

Lynis не дає абсолютної оцінки “безпечний” або “небезпечний”. Він показує, наскільки система відповідає перевіреним правилам і рекомендаціям. Для виробничих систем зазвичай орієнтуються на 80+ балів, тоді як нижчі значення вказують на потребу в додаткових змінах.

Водночас цей показник не варто трактувати як єдиний критерій безпеки. Дві системи з однаковим Hardening Index можуть мати різний реальний рівень захищеності, якщо вони виконують різні функції або мають різну модель ризиків.

## 2.4. Межі та обмеження методології дослідження

Будь-яке коректне дослідження має не лише описувати свої сильні сторони, а й чесно окреслювати обмеження. У цьому підрозділі наведено основні межі застосованої методології.

### Тестове середовище замість production

Усі вимірювання виконуються на чистій віртуальній машині з Ubuntu 24.04 LTS без реального трафіку та без прикладного застосунку. Це свідоме спрощення, бо на production-сервері частина змін може впливати на роботу сайту або суміжних сервісів. Наприклад, жорсткий chroot в FTP може бути несумісним із логікою деяких CMS, а зміна порту SSH потребує окремого оновлення конфігурацій доступу.

Саме тому деякі зміни в роботі скрипта реалізуються обережно, а частина конфігурацій супроводжується попередженнями для адміністратора.

### Обмеження Hardening Index як метрики

Hardening Index Lynis показує, наскільки система відповідає відомим перевіркам, але не гарантує повної відсутності невідомих вразливостей. Система з високим балом усе одно може бути вразливою до 0-day, яких Lynis просто не перевіряє. Тому в межах цієї роботи Hardening Index розглядається як індикатор покращення конфігурації, а не як остаточна оцінка захищеності.

### Область застосування скрипта

Розроблений інструмент орієнтований на один конкретний сценарій: один сервер, Ubuntu 24.04 LTS і типовий LAMP/LEMP-стек. Він не призначений для мультисерверних інфраструктур, нестандартних конфігурацій, частково вже захардкожених систем або контейнерних середовищ, де модель безпеки суттєво відрізняється від звичайної VM.

Ці обмеження не знижують практичну цінність інструменту, але їх потрібно враховувати, щоб правильно інтерпретувати результати дослідження.
